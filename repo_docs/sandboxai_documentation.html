# Purpose and Scope The repository is part of the **sandboxai** project, which aims to provide a secure, sandboxed environment for executing arbitrary code or commands. The system enables users to interact with this sandbox through APIs and clients, ensuring isolation and safety when running untrusted code. This aligns with the broader goal of creating tools for evaluating or testing code in controlled conditions. # Core Features - **Sandbox Execution**: Provides a secure containerized environment for executing code. - **API Integration**: Exposes RESTful APIs (v1) for programmatic interaction with the sandbox. - **Client Libraries**: Includes both Python and Go client libraries for easier integration into various workflows. - **End-to-End Testing**: Comprehensive test suites in `test/e2e` ensure reliability and correctness. - **Extensibility**: Experimental modules like `crewai.py` suggest ongoing development for advanced use cases. # Target Audience/Users The primary users of this repository are: - Developers needing secure environments for code execution. - Researchers or engineers working on automated testing or evaluation systems. - Teams integrating sandboxed execution capabilities into larger applications. # Main Technologies or Frameworks Used - **Go**: Used extensively for building the backend services, including the sandbox handler and API generation. - **Python**: Provides client libraries and scripts for interacting with the sandbox. - **Docker**: Integral to sandboxing, as evident from files like `box.Dockerfile`. - **Excalidraw**: Used for diagramming the architecture (e.g., `overview.excalidraw.png`). - **OpenAPI**: API definitions follow OpenAPI standards (`api/v1.yaml`). This repository integrates these technologies to deliver a robust solution for secure code execution.

# System Architecture Overview This repository, named "sandboxai," is designed to create a secure and isolated environment for executing code snippets or commands. The system is built with modularity in mind, allowing different components to interact seamlessly while maintaining clear separation of responsibilities. Below is a detailed breakdown of the architecture. ## High-Level Description The sandboxai system provides an API-driven interface to execute arbitrary code or shell commands within a controlled and isolated environment. It leverages both Go and Python languages for backend services and client interactions, ensuring flexibility and scalability. The architecture includes Docker-based sandboxing mechanisms to ensure safety and reproducibility of executions. --- ## Main Components and Their Roles * **Go Backend**: This component houses the core logic of the application, including the API server and sandbox handler. It manages incoming requests, validates them, and orchestrates the execution process. The go/sandboxaid/main.go file serves as the entry point for the backend service. It interacts with the Docker daemon to manage containers that act as sandboxes. * **Python Client**: Written in Python, this component acts as the primary interface for users who wish to interact with the sandboxai system. It communicates with the Go backend via RESTful APIs defined in api/v1.yaml. The python/sandboxai/client/v1.py file contains the implementation for making API calls to the backend. * **Docker Integration**: Docker is used extensively to create isolated environments (containers) where user-submitted code or commands are executed. These containers ensure that malicious or faulty code cannot affect the host system. Configuration details for the Docker setup can be found in box.Dockerfile and go/sandboxaid/client/docker/docker.go. * **API Specification**: A well-defined API specification outlines the endpoints and data formats expected by the system. This ensures consistency across clients and servers. The api/v1.yaml file defines the structure of the API, which includes methods for submitting jobs, retrieving results, and managing resources. * **Test Suite**: Comprehensive testing is implemented at multiple levels to ensure reliability and correctness. End-to-end tests (located in test/e2e and go/test/e2e) simulate real-world scenarios and verify the entire workflow from request submission to result retrieval. * **Documentation**: Clear documentation is provided to guide developers and users through the system's functionality. Diagrams such as docs/diagrams/overview.excalidraw.png visually represent the architecture, while files like README.md offer textual explanations. * **Experimental Features**: Some experimental features are housed under the python/sandboxai/experimental directory. For example, crewai.py might contain prototypes or new ideas being explored but not yet integrated into the main system. --- ## Data Flow Between Components 1. A user submits a request through the Python client, specifying the code or command they want to execute along with any necessary parameters. 2. The Python client sends this request to the Go backend using the predefined API endpoints. 3. Upon receiving the request, the Go backend validates it and forwards the task to the sandbox handler. 4. The sandbox handler creates a new Docker container configured according to the requirements specified in the request. 5. Inside the container, the requested code or command is executed in isolation. 6. Once execution completes, the output (or error messages) is captured and returned to the Go backend. 7. Finally, the Go backend relays the result back to the Python client, which presents it to the user. --- ## External Dependencies and Integrations * **Docker**: Essential for creating isolated execution environments. Without Docker, the system would lack the ability to safely run untrusted code. * **RESTful APIs**: Used for communication between the Python client and the Go backend. This choice allows flexibility and compatibility with various programming languages. * **YAML Files**: Employed for defining API specifications and configuration settings, ensuring clarity and maintainability. * **Markdown and PNG Files**: Utilized for documenting and illustrating the system architecture, helping onboard new contributors and users. By combining these elements, sandboxai achieves its goal of providing a robust, secure, and user-friendly platform for executing code in a controlled manner.

1. **API Layer**: Defined in `go/api/v1`, this layer exposes RESTful endpoints for interacting with the system. 2. **Client Libraries**: Located in `go/client/v1` and `python/sandboxai/client/v1.py`, these libraries allow clients to interact with the API. 3. **Core Service**: Implemented in `go/sandboxaid`, this service handles core business logic and communicates with the Docker client. 4. **Test Suite**: Comprehensive test cases are located in `test/e2e` and `go/test/e2e`, ensuring robustness.

Communication Flow

- The client sends a request to the API. - The API forwards the request to the `sandboxaid` service. - The service processes the request, optionally invoking Docker operations. - The result is returned through the API to the client. ---
